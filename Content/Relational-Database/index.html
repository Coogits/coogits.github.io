    Relational Database

    A Relational Database (RDB) is a structured collection of data organized into tables that are connected through relationships based on keys.
    It follows the principles of Relational Model, introduced by Edgar F. Codd in 1970.

    Concept

    Tables

    Data is stored in rows (tuples) and columns (attributes). $$\begin{array}{|c|c|c|}\hline \rowcolor{gray} \text{UserID}&\text{FirstName}&\text{LastName}\\\hline \text{1}&\text{Isaac}&\text{Newton}\\\hline \text{2}&\text{Galileo}&\text{Galilei}\\\hline \end{array}$$

    • A table (relation) represents a collection of related data.
    • Attributes (columns) define the properties of an entity.
    • Tuples (rows) store individual records of data.

    Entity

    An entity represents a real-world object or concept that is modeled in the database.

    • In the relational model, an entity type corresponds to a table.
    • Each entity has a unique identifier (such as a primary key).
    • Example: The Users table represents the User entity.

    Attribute

    An attribute is a property or characteristic of an entity, represented as a column in a table.

    • Attributes define the structure of a relation by specifying the type of data stored.
    • Each attribute has a domain (a set of valid values).
    • Example: In the Users table, FirstName and LastName are attributes that store user names.

    Tuple

    A tuple (Record) is a single, ordered set of values corresponding to attributes in a table.

    • Each row in a table is a tuple, representing a unique data entry.
    • A tuple consists of attribute values that describe an entity instance.
    • Example: A row in the Users table represents a specific user.

    Instance

    A database instance refers to the actual stored data at a specific point in time.

    • The relation instance consists of all the tuples currently in a table.
    • Each tuple represents a specific occurrence (instance) of an entity within an entity set.
    • As data changes, the database instance evolves while the schema remains unchanged.

    Keys

    • Primary Key (PK):
      Uniquely identifies each row. $$\begin{array}{|c|c|c|}\hline \rowcolor{gray} \text{UserID(PK)}&\text{FirstName}&\text{LastName}\\\hline \text{1}&\text{Isaac}&\text{Newton}\\\hline \text{2}&\text{Galileo}&\text{Galilei}\\\hline \end{array}$$

      • Cannot contain NULL values.

    • Foreign Key (FK):
      Establishes relationships between tables: $$\begin{array}{|c|c|c|}\hline \rowcolor{gray} \text{OrderID(PK)}&\text{OrderDate}&\text{UserID(FK)}\\\hline \text{101}&\text{10/11}&\text{1}\\\hline \text{102}&\text{11/11}&\text{2}\\\hline \end{array}$$

      • A field in one table that references the Primary Key of another table.

    • Composite Key:
      A Primary Key made up of multiple columns: $$\begin{array}{|c|c|c|}\hline \rowcolor{gray} \text{UserID(FK)}&\text{DegreeID(FK)}&\text{YearErned}\\\hline \text{1}&\text{101}&\text{2004}\\\hline \text{2}&\text{102}&\text{2015}\\\hline \end{array}$$

      • Used when a single column cannot uniquely identify a row.

    • Surrogate Key:
      A system-generated key that has no real-world meaning. $$\begin{array}{|c|c|c|c|}\hline \rowcolor{gray} \text{UserID}&\text{FirstName}&\text{LastName}&\text{IDðŸ”‘}\\\hline \text{}&\text{Isaac}&\text{Newton}&\text{1}\\\hline \text{2}&\text{Galileo}&\text{Galilei}\\\hline \end{array}$$

      • Used when a natural unique identifier does not exist or is inefficient.

    Others

    • Data Integrity:
      Ensures accuracy with constraints (e.g., UNIQUE, NOT NULL).
    • Normalization:
      Reduces redundancy by organizing data efficiently.
    • Structured Query Language (SQL):
      Used to manage and manipulate data.

    Relationship

    In a relational database, a relationship defines how two or more tables (entities) are logically connected based on their key attributes. Relationships establish associations between tables to ensure data integrity and meaningful connections between different entity types.

    Types of Relationships:

    • One-to-One (1:1)
      • Each record in Table A is related to one and only one record in Table B, and vice versa.
      • Example: A User has one Passport.
    • One-to-Many (1:M)
      • A single record in Table A can relate to multiple records in Table B, but each record in Table B relates to only one record in Table A.
      • Example: A Customer can place multiple Orders, but each Order is linked to a single Customer.
    • Many-to-Many (M:N)
      • Multiple records in Table A can relate to multiple records in Table B.
      • Requires a junction (bridge) table with foreign keys from both tables.
      • Example: Students enroll in multiple Courses, and each Course has multiple enrolled Students.

    Schema

    Two Examples of Relational Database Schemas

    • The table name (e.g., Customer, Order) represents the entity.
    • The columns represent the attributes of the entity (e.g. Name).
    • Each row represents a tuple (record) in the entity (e.g. the row of Rebecca Philips).
    • The CustomerID in the Customer table is a Primary Key (PK).
    • The CustomerID in the Order table is a Foreign Key (FK), linking orders to customers.

    Relational Database Management Systems (RDBMS)

    RDBMSs use the relational model, where data is structured in tables with rows (tuples) and columns (attributes).

    Some of the most widely used RDBMS include:

    • Oracle Database
    • MySQL
    • PostgreSQL
    • Microsoft SQL Server
    • SQLite
    • MariaDB
