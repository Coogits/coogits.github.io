    Logical Model (Database)

    The Logical Model defines how data is structured, related, and constrained while remaining independent of a specific database management system (DBMS). It serves as an intermediary step between the Conceptual Model and the Physical Model, ensuring that data is logically organized before being implemented in a specific system.

    Key Characteristics

    Data Structure Definition

    • Identifies entities (real-world objects/concepts).
    • Specifies attributes (properties of entities).
    • Assigns primary keys (PK) to uniquely identify records.
    • Defines foreign keys (FK) to enforce relationships between entities.

    Relationship Representation

    Establishes relationships between entities:

    • One-to-One (1:1) – Each entity instance relates to a single instance of another entity.
    • One-to-Many (1:M) – A single entity instance can relate to multiple instances of another entity.
    • Many-to-Many (M:N) – Multiple instances of one entity relate to multiple instances of another entity.

    Normalization

    • Removes redundancy by organizing data into separate tables.
    • Follows normalization rules (1NF, 2NF, 3NF, BCNF) to ensure data integrity.
    • Avoids unnecessary duplication of data.

    Constraints & Integrity Rules

    • Entity Integrity – Each table must have a unique primary key.
    • Referential Integrity – Foreign keys must reference valid primary keys.
    • Domain Constraints – Attributes must contain valid values (e.g., price must be positive).
    • Business Rules – Custom constraints based on real-world logic.

    Relational Schema Diagram

    Plain Text

    CUSTOMER (CustomerID PK, Name, Email, PhoneNumber)
ORDER (OrderID PK, OrderDate, CustomerID FK)
PRODUCT (ProductID PK, Name, Price, Category)
ORDER_DETAILS (OrderID FK, ProductID FK, Quantity, Subtotal)

    Relationships:

    • CustomerID in ORDER references CustomerID in CUSTOMER (1:M relationship).
    • OrderID in ORDER_DETAILS references OrderID in ORDER (M:1 relationship).
    • ProductID in ORDER_DETAILS references ProductID in PRODUCT (M:1 relationship).
    • ORDER_DETAILS serves as a junction table for the M:N relationship between ORDER and PRODUCT.
