    Database Model

    A database model defines how data is structured, stored, and managed in a database. There are different levels of abstraction when designing a database, primarily categorized into Conceptual, Logical, and Physical models.

    1. Conceptual Model → High-level, focuses on business requirements.
    2. Logical Model → More structured, defines attributes, keys, and relationships.
    3. Physical Model → Implements the logical design using a specific DBMS.

    Below, we focus on the Conceptual and Logical models as part of the database design process.

    Conceptual Model

    Defines "what" the system represents.

    • High-level representation of data and relationships.
    • Prototypes Data (high-level representation of entities, attributes, and relationships).
    • Focuses on business requirements, not technical implementation.
    • Independent of DBMS (no tables, columns, or specific data types).
    • Uses Entity-Relationship Diagrams (ERD) with entities, attributes, and relationships.
    Example (Conceptual Model)
    • Entities: Customer, Order, Product.
    • Relationships:
      • A Customer places multiple Orders.
      • An Order contains multiple Products.

    Logical Model

    Specifies "how" data is structured.

    • More detailed than conceptual but still DBMS-independent.
    • Defines Data (detailed structure with primary keys, foreign keys, and normalization).
    • No physical storage details (e.g., indexing, partitions).
    • Normalization is applied to reduce redundancy.
    Example (Logical Model)
    • Entities & Attributes:
      • Customer (CustomerID, Name, Email).
      • Order (OrderID, OrderDate, CustomerID).
      • Product (ProductID, Name, Price).
      • OrderDetails (OrderID, ProductID, Quantity).
    • Relationships:
      • CustomerID is a foreign key in Order.
      • OrderID & ProductID are foreign keys in OrderDetails.

    Physical Model

    Details "how" the database is implemented.

    • Defines tables, indexes, storage details, partitions, etc.
    • Tailored to a specific database system (e.g., MySQL, PostgreSQL).
    • Considers performance, security, and optimization.

    Summary

    Feature Conceptual Model Logical Model Physical Model
    Abstraction High (business-focused) Detailed (database-focused) Low (storage-focused)
    Focus Data concepts & relationships Data structure & constraints Storage, performance, and optimization
    DBMS-Dependent? No No Yes
    Level of Detail High-level Medium-level Low-level (implementation details)
    Key Components Entities, Attributes, Relationships Tables, Keys, Constraints Storage, Indexing, Partitions
    Diagram Type ER Diagram Relational Schema Diagram Physical Schema
    Normalization Not applied Applied to reduce redundancy Considered for performance
